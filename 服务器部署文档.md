## 一、准备

### 1.说明

该服务器为分布式集群架构。

在下面 **环境部署** 中，除java环境之外，其他中间件(zookeeper,mongodb,redis,kafka,nginx) **建议**使用云服务器平台托管服务。

如果使用ec2服务器搭建的中间件服务，则最大同时在线人数最好**不要超过 1w**。

如果使用托管服务，则在 **环境部署** 步骤中可跳过 **中间件服务环境**



### 2.规范

以下关于固定目录说明时，假设 `/` 目录挂载磁盘拥有足够的存储空间

#### 1.docker 目录

根目录在 `/data/docker`， 里面有两个子文件夹 `conf` 和 `volum`

​	 `conf` 对应的是`docker` 存储数据的根目录

​	 `volum` 对应的是`docker`容器各自的映射目录



#### 2.游戏目录

根目录在 `/data/game`， 里面创建各自服务的目录

为了游戏能够不停服更新，需要创建两个文件夹。

**例：**

部署`hall`服务时,在 `/data/game` 目录下创建 `hall1`, `hall2` 目录，这两个目录中只有 `application-master.yaml` 里面的`cluster.name`和`cluster.tcpAddress`不一样，其他配置全部相同

如果还要部署更多的`hall`服务，建议部署在不同的`ec2`上，则同样创建项目目录`hall3`,`hall4`......



#### 3.游戏服务节点命名规则

1.**同一集群**内的节点名称**不可重复**
2.同一集群内的节点名称不可出现字符串包含现象。
	**错误**举例：
      A节点名称: TEST_HALL
      B节点名称: TEST_HALL1
   上面B的名称中已经包含的A节点的名称，这是错误的。

​    **正确**举例：
​       A节点名称: TEST_HALL1
​       B节点名称: TEST_HALL2



#### 4.时区

如果对服务器时间有时区要求，可先修改系统时区后，然后重启云服务器



#### 5.清除日志定时任务

为了避免服务器日志的无限增长，导致占满磁盘，所以**一定**要建立定时任务来清理日志。

可使用操作系统的cron或者timer来做定时执行该脚本的任务，这里不过多描述

##### 4.脚本(源服务器)

执行时若遇到权限问题，自行解决

```shell
#!/bin/sh

#通用配置
REMOTE_HOST="172.31.33.78"
DATE=$(date +%Y%m%d_%H%M%S)

# redis需要备份的目录
REDIS_BACKUP_DIR="/data/docker/volum/redis01/data/"
# 远程的备份目标目录
REDIS_REMOTE_DIR="/data/backup/redis01/"

#同步redis数据
RSYNC_REDIS_CMD="rsync -avz --exclude='temp-*' ${REDIS_BACKUP_DIR} ec2-user@${REMOTE_HOST}:${REDIS_REMOTE_DIR}"
# 执行rsync命令，并进行重定向
$RSYNC_REDIS_CMD > /dev/null 2>&1

echo '同步redis结束'

# mongo配置
MONGO_CONTAINER="mongo01"
# 先将备份文件导出到本地的目录
MONGO_BACKUP_DIR="/data/docker/volum/mongo01/backup"
# 远程的备份目标目录
MONGO_REMOTE_DIR="/data/backup/mongo01"
MONGO_PWD="123456"
# 容器内临时存储备份文件
CONTAINR_TMP_BACKUP_NAME="/data/backup/latest_backup"

# 清理容器内旧备份
sudo docker exec ${MONGO_CONTAINER} rm -rf ${CONTAINR_TMP_BACKUP_NAME}/* 2>/dev/null

# 删除宿主机上的旧备份
sudo rm -rf ${MONGO_BACKUP_DIR}/*

# 删除远程服务器上的旧备份
ssh ec2-user@${REMOTE_HOST} "rm -rf ${MONGO_REMOTE_DIR}/*"

# 执行备份
sudo docker exec ${MONGO_CONTAINER} mongodump \
    --host localhost \
    --port 27017 \
    --username admin \
    --password ${MONGO_PWD} \
    --out ${CONTAINR_TMP_BACKUP_NAME} \
    --gzip \
    --quiet

# 复制到宿主机
sudo docker cp ${MONGO_CONTAINER}:${CONTAINR_TMP_BACKUP_NAME} ${MONGO_BACKUP_DIR}/

# 同步到远程服务器
RSYNC_MONGO_CMD="rsync -avz ${MONGO_BACKUP_DIR}/ ec2-user@${REMOTE_HOST}:${MONGO_REMOTE_DIR}/"
# 执行rsync命令，并进行重定向
$RSYNC_MONGO_CMD > /dev/null 2>&1

echo '同步mongodb结束'
```

该目录存放的是该服务的基础配置，如数据库连接，日志配置等

`application.yaml`    该文件决定本服务使用的是哪个配置文件

```shell
spring.profiles.active:
  master  #表示使用的是application-master配置文件
```

`application-master.yaml` 该文件配置了服务的端口或者数据库连接等等

`logback.xml` 该文件为日志配置文件

`nodeConfig.json` 该文件中可配置节点权重和白名单

```shell
{
  "weight": 1,  #节点权重，值范围 >=0。正常运行时要 >=1。如果为0，则该节点若没有连接时，会自动关闭
  "whiteIpList":["192.168.3.90"], #白名单ip，(在gate,gm,recharge中设置无效)
  "whiteIdList": ["1300001"]  #白名单id，(在gate,hall,gm,recharge中设置无效)
}
```

## 五、服务器更新

根据服务种类的来分，更新可以分为3种不同的类型

### 第一种

`gate`、`account` 和 `recharge`  这一类带有http请求，并且由 `nginx` 代理的服务

需要将 nginx 配置中的 `weight` 改为 **0**  或者将该条配置注释掉，等待没有请求后，直接执行 kill 命令(不能 kill -9) 关闭即可



### 第二种

`hall`、`slots`、`table` 和 `poker` 这一类游戏逻辑服

更新举例：

1.假设当前正在 `hall1` 目录运行大厅服务，并且 `hall2`处于未执行状态

2.可在`hall2`目录更新到最新的代码，(可在`nodeConfig.json`中添加白名单，那么启动后只有在白名单内的ip才会进入到该服务，可以验证功能是否更新)，然后启动，（验证功能后，在后台 **游戏设置 -> 服务器列表**，找到`hall2`编辑，清除白名单即可）

3.通过后台的 **游戏设置 -> 服务器列表**  找到`hall1`编辑，将权重设置为 **0**

4.当`hall1` 连接退出完毕后，`hall1`会自动结束进程



### 第三种

`gm` 服务，可直接 `kill` 进程，更新后重启即可

针对所有的服务，如果是 kill <PID> 结束进程，需要等待10秒后再启动。



**注意事项**：

若非必要，禁止在凌晨 0-5 点(系统时间)更新服务器，因为这时候服务器可能在执行定时任务



## 六、问题排查

遇到服务挂掉的情况

#### 1.若该进程不存在

则打包该进程目录下的 `log` 文件夹交由开发人员



#### 2.若该进程存在

​	假设是`hall` 挂掉，或者是因为该进程导致`cpu`或者内存被打满的情况

​	首先找到该进程的`pid` , 假设为 `1234`

​	则执行命令 1

```shell
jstack 1234 > jstack1234.txt  #将1234替换为正确的pid
```

​	继续执行命令2

```shell
jmap -dump:live,format=b,file=jmap1234.hprof 1234   #将1234替换为正确的pid
```

注意：**命令1** 和 **命令2** 会对该进程有性能影响，只能在该服务不可用的情况下使用



然后将上面的 `jstack1234.txt`,`jmap1234.hprof` 和 `log` 文件一起打包交由开发人员
